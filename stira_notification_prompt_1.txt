app opens for the first time each day. Nothing else fires. No background

workmanager. Zero Firebase Cloud Functions. Zero Blaze plan required. Every
notification fires from the device itself using on-device data from the existing
Hive boxes: stira_prefs and stira_logs.

history, calculates their actual vulnerable time windows from data, tracks urge
velocity, monitors location for danger zones, and sends notifications that feel
like the app understands the user — not generic reminders. Notifications must
also be real-time reactive: when the user submits a check-in inside the app, the
system immediately responds, cancels irrelevant pending notifications, and
schedules the right follow-up.

This file contains a single class StiraBehaviourReader with a single static

The StiraBehaviourSnapshot object must contain these fields:

  int currentStreak          — from stira_prefs
  int totalCleanDays         — from stira_prefs
  int hoursSinceLastCheckIn  — calculated from stira_prefs last_checkin_at
  bool checkedInToday        — true if last_checkin_at is today's date
  int sentToday              — from stira_prefs, reset to 0 at midnight
  DateTime? lastSentAt       — from stira_prefs
  bool restMode              — from stira_prefs, true after 5 ignored notifs
  bool dopamineJournalToday  — check stira_logs for a journal entry today
  int peakHour               — calculated from check-in history (see below)
  int peakHourConfidence     — 0 to 100, how confident the peak hour is
  double urgeVelocity        — calculated from last 3 check-ins (see below)
  String topTrigger          — most frequent trigger across all check-ins
  double stabilityTrend      — positive means improving, negative means worse
  String coreNeed            — from stira_prefs, set during onboarding
  bool sameTimeTomorrowDue   — true if a commitment exists from 23-25h ago
  String commitmentText      — the commitment text if sameTimeTomorrowDue

  int daysSinceInstall       — calculated from stira_prefs install_date
  int personalBestStreak     — from stira_prefs

HOW TO CALCULATE peakHour:
Read all check-in records from stira_logs where urge level is 6 or higher.
Count how many high-intensity check-ins occurred at each hour of the day (0-23).
The peakHour is the hour with the highest count. If there are fewer than 5
total high-intensity check-ins, fall back to the peak_window value from onboarding:
Morning = hour 9, Afternoon = hour 14, Evening = hour 19, Night = hour 22.
peakHourConfidence is: (count at peakHour / total high-intensity check-ins * 100)
clamped between 0 and 100. If using onboarding fallback, confidence is 30.

HOW TO CALCULATE urgeVelocity:
Read the last 3 check-in records from stira_logs sorted by timestamp descending.
urgeVelocity = urgeLevel[0] - urgeLevel[2] (newest minus oldest of the 3).
A positive number means urges are rising. A value of 3 or higher is critical.
If fewer than 3 check-ins exist, urgeVelocity is 0.

HOW TO CALCULATE topTrigger:
Read all check-in records from stira_logs. For each check-in, read the triggers
list. Count occurrences of each trigger string. The topTrigger is the string
that appears most often. If no check-ins exist, use the first value from the
primary_triggers field in stira_prefs.

HOW TO CALCULATE stabilityTrend:
Compare average urge intensity of the last 7 days versus the 7 days before that.
stabilityTrend = avgUrge(days8to14) - avgUrge(days1to7).
A positive stabilityTrend means the user is improving (lower urges recently).
A negative stabilityTrend means urges are getting worse.

HOW TO CHECK sameTimeTomorrowDue:
Read from stira_logs all entries with type "commitment". Find the most recent
one. If its saved_at timestamp is between 23 and 25 hours ago, and it has no
"outcome" field set, then sameTimeTomorrowDue is true and commitmentText is its
"text" field.

Create a new file: lib/services/stira_throttle_manager.dart

This class enforces all limits on notification sending. Every notification in
the entire system must call ThrottleManager.canSend() before firing. If
canSend() returns false, the notification is silently dropped.

RULES canSend() must enforce:

QUIET HOURS RULE: If the current hour is 22 or later, or earlier than 8, return
false. Crisis override: if isCrisis is true, quiet hours are ignored.

DAILY LIMIT RULE: Read sentToday from stira_prefs. If sentToday is 2 or more
and restMode is false, return false. If sentToday is 1 or more and restMode is
true, return false. If crisisActive is true, the limit increases to 4.

MINIMUM GAP RULE: Read lastSentAt from stira_prefs. If the difference between
now and lastSentAt is less than 3 hours, return false. Crisis override: if
isCrisis is true, minimum gap is ignored.

MIDNIGHT RESET: At the start of canSend(), check if the stored last_reset_date
in stira_prefs matches today's date (YYYY-MM-DD format). If it does not match,
reset sentToday to 0 and update last_reset_date. This ensures the daily limit
resets every midnight automatically.

REST MODE CHECK: Read rest_mode_until from stira_prefs. If it exists and is in
the future, restMode is true. If it has passed, clear it and set restMode false.

METHODS TO IMPLEMENT:

canSend({bool isCrisis = false}) → bool: applies all rules above, returns true
if a notification can be sent right now.

recordSent(): increments sentToday in stira_prefs by 1, updates lastSentAt to
now.

recordTap(): resets consecutive_untapped in stira_prefs to 0. If rest_mode was
active, clears it.

recordUntapped(): increments consecutive_untapped in stira_prefs by 1. If
consecutive_untapped reaches 5, sets rest_mode_until to 7 days from now.

setCrisisActive(bool active): writes crisis_active to stira_prefs.

STEP 7 — CREATE THE INTELLIGENCE ENGINE (THE BRAIN)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create a new file: lib/services/stira_intelligence_engine.dart

This file has two responsibilities:
1. A background cycle that runs every 15 minutes using workmanager.
2. Real-time reaction methods that are called directly from the UI the moment
   the user does something.

BACKGROUND ENTRY POINT:
Create a top-level function called callbackDispatcher() marked with
@pragma("vm:entry-point"). This function calls Workmanager().executeTask().
Inside the task handler, check the task name. If it matches
"stira_intelligence_cycle", open all required Hive boxes, initialize
NotificationService, and call StiraIntelligenceEngine.runCycle(). If the task
name matches "stira_location_check", call StiraLocationMonitor.checkLocation().
Always return Future.value(true) at the end regardless of what happened.

This callbackDispatcher function must be at the top level of the file, not
inside any class.

START BACKGROUND CYCLE:
Add a static method startBackgroundCycle() that calls:
Workmanager().initialize(callbackDispatcher, isInDebugMode: false)
Then registers a periodic task with:
  uniqueName: "stira_intelligence_task"
  taskName: "stira_intelligence_cycle"
  frequency: Duration(minutes: 15)
  constraints: Constraints(networkType: NetworkType.not_required,
    requiresBatteryNotLow: false, requiresDeviceIdle: false)
  existingWorkPolicy: ExistingWorkPolicy.keep

THE DECISION CYCLE — runCycle():
This method runs every 15 minutes. It reads the behaviour snapshot, then works
through the following decision steps in exact order. It fires at most one
notification per cycle (the first one that passes all conditions) and then
returns. One notification per 15-minute cycle maximum.

Step 1 — MIDNIGHT RESET
Call ThrottleManager.canSend() which handles the midnight sentToday reset
automatically. No additional code needed here.

Step 2 — CRISIS: URGE VELOCITY CHECK
Read the snapshot. If snapshot.urgeVelocity >= 3.0 and snapshot.checkedInToday
is true: this means the user has checked in multiple times today and their
urges are rising fast. Fire immediately without throttle check.
  Call NotificationService.showNow with:
    id: ID_URGE_VELOCITY
    title and body from Messages.urgeVelocity(snapshot)
    channelId: "stira_critical"
  Call ThrottleManager.recordSent().
  Return. Do not check any other conditions this cycle.

Step 3 — MILESTONE: STREAK MILESTONES
Read snapshot.currentStreak. Check if it matches any of these: 3, 7, 14, 30,
60, 90, 180, 365. If it matches, read last_milestone_fired from stira_prefs.
If last_milestone_fired does not equal the current streak value, this milestone
has not been announced yet. Fire it:
  Call NotificationService.showNow with:
    id: ID_STREAK_MILESTONE
    title and body from Messages.streakMilestone(snapshot.currentStreak, snapshot.userName)
    channelId: "stira_high"
  Write current streak value to last_milestone_fired in stira_prefs.
  Call ThrottleManager.recordSent().
  Return.
This fires regardless of quiet hours and daily limit because milestones must
never be blocked.

Step 4 — MILESTONE: PRE-MILESTONE TOMORROW
If snapshot.currentStreak + 1 matches any of: 3, 7, 14, 30, 60, 90, 180, 365
AND the current hour is between 19 and 21 (7 PM to 9 PM):
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_PRE_MILESTONE
      title and body from Messages.preMilestone(snapshot.currentStreak + 1)
      channelId: "stira_high"
    Call ThrottleManager.recordSent().
    Return.

Step 5 — SAME TIME TOMORROW COMMITMENT
If snapshot.sameTimeTomorrowDue is true:
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_SAME_TIME_TOMORROW
      title and body from Messages.sameTimeTomorrow(snapshot.userName, snapshot.commitmentText)
      channelId: "stira_high"
    Call ThrottleManager.recordSent().
    Return.

Step 6 — PREDICTIVE: VULNERABILITY WARNING
Calculate minutesUntilPeakHour = (snapshot.peakHour * 60) - (currentHour * 60 + currentMinute).
If minutesUntilPeakHour is between 30 and 50 AND snapshot.peakHourConfidence >= 45:
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_VULNERABILITY_WARNING
      title and body from Messages.vulnerabilityWarning(snapshot)
      channelId: "stira_high"
    Call ThrottleManager.recordSent().
    Return.

Step 7 — CHECK-IN REMINDER
If snapshot.checkedInToday is false AND snapshot.hoursSinceLastCheckIn < 48:
  Calculate minutesToPeak = abs((snapshot.peakHour - currentHour) * 60).
  If minutesToPeak <= 90 (within 1.5 hours of peak window):
    If ThrottleManager.canSend() returns true:
      Call NotificationService.showNow with:
        id: ID_CHECKIN_REMINDER
        title and body from Messages.checkInReminder(snapshot)
        channelId: "stira_high"
      Call ThrottleManager.recordSent().
      Return.

Step 8 — DISENGAGEMENT
If snapshot.hoursSinceLastCheckIn >= 72:
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_DISENGAGEMENT_72H
      title and body from Messages.disengagement72h(snapshot)
      channelId: "stira_low"
    Call ThrottleManager.recordSent().
    Return.
If snapshot.hoursSinceLastCheckIn >= 48:
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_DISENGAGEMENT_48H
      title and body from Messages.disengagement48h(snapshot)
      channelId: "stira_low"
    Call ThrottleManager.recordSent().
    Return.

Step 9 — DOPAMINE JOURNAL
If snapshot.dopamineJournalToday is false AND current hour is 21:
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_DOPAMINE_JOURNAL
      title and body from Messages.dopamineJournal()
      channelId: "stira_low"
    Call ThrottleManager.recordSent().
    Return.

Step 10 — IDENTITY PUSH
If snapshot.sentToday == 0 AND current hour == snapshot.peakHour:
  If ThrottleManager.canSend() returns true:
    Call NotificationService.showNow with:
      id: ID_IDENTITY_PUSH
      title and body from Messages.identityPush(snapshot.daysSinceInstall)
      channelId: "stira_low"
    Call ThrottleManager.recordSent().
    Return.

Step 11 — WEEKLY REPORT
If today is Sunday AND current hour is 8:
  Read last_weekly_report_sent from stira_prefs. If it does not match today's
  date string:
    If ThrottleManager.canSend() returns true:
      Call NotificationService.showNow with:
        id: ID_WEEKLY_REPORT
        title and body from Messages.weeklyReport()
        channelId: "stira_low"
      Write today's date to last_weekly_report_sent in stira_prefs.
      Call ThrottleManager.recordSent().
      Return.

Step 12 — STABILITY IMPROVING
If snapshot.stabilityTrend > 0.5 (user is measurably improving):
  Read last_stability_notif from stira_prefs. If it was more than 7 days ago
  or has never been sent:
    If ThrottleManager.canSend() returns true:
      Call NotificationService.showNow with:
        id: ID_STABILITY_UP
        title and body from Messages.stabilityImproving(snapshot)
        channelId: "stira_low"
      Write today's date to last_stability_notif in stira_prefs.
      Call ThrottleManager.recordSent().
      Return.

If none of the above conditions triggered anything, the cycle ends silently.
This is correct. Silence on stable days is intentional.


REAL-TIME REACTION METHOD: reactToAction(UserAction action)
This static method is called directly from the UI code the moment the user does
something. It must be fast — it runs on the main thread. Call
NotificationService.initialize() at the start in case it was not already called.

When action == UserAction.checkInSubmitted:
  1. Cancel these pending notifications immediately:
     ID_CHECKIN_REMINDER, ID_DISENGAGEMENT_48H, ID_DISENGAGEMENT_72H
  2. Read the most recently saved check-in from stira_logs to get the urge level.
  3. If urge level >= 8:
     - Call ThrottleManager.setCrisisActive(true)
     - Call NotificationService.fireSOSHeartbeat()
  4. If urge level < 8:
     - Call ThrottleManager.setCrisisActive(false)
  5. Read snapshot. Schedule a follow-up check-in notification for 4 hours from
     now using NotificationService.scheduleAt with:
       id: ID_CHECKIN_FOLLOWUP
       title and body from Messages.checkInFollowUp(snapshot)
       when: DateTime.now().add(Duration(hours: 4))
       channelId: "stira_low"
       payload: "/checkin"

When action == UserAction.relapseLogged:
  1. Write DateTime.now() to last_relapse_at in stira_prefs.
  2. Read snapshot.
  3. Call NotificationService.showNow immediately (no throttle check, crisis):
       id: ID_RELAPSE_SUPPORT
       title and body from Messages.relapseSupport(snapshot)
       channelId: "stira_critical"
       payload: "/home"
  4. Call ThrottleManager.recordSent().
  5. Call ThrottleManager.setCrisisActive(false).

When action == UserAction.commitmentSaved:
  Read the most recent commitment text from stira_logs.
  Read snapshot.
  Call NotificationService.scheduleAt with:
    id: ID_SAME_TIME_TOMORROW
    title and body from Messages.sameTimeTomorrow(snapshot.userName, commitmentText)
    when: DateTime.now().add(Duration(hours: 24))
    channelId: "stira_high"
    payload: "/checkin"

When action == UserAction.appOpened:
  Cancel ID_DISENGAGEMENT_48H and ID_DISENGAGEMENT_72H.
  These fire only when the user is away. When they open the app, cancel them.

When action == UserAction.notificationTapped:
  Call ThrottleManager.recordTap().

ENUM UserAction:
Define this enum at the bottom of the file:
enum UserAction {
  checkInSubmitted,
  relapseLogged,
  commitmentSaved,
  appOpened,
  notificationTapped,
}

STEP 8 — CREATE STIRA LOCATION MONITOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create a new file: lib/services/stira_location_monitor.dart

This file monitors location in the background and fires when the user enters a
danger zone or safe zone.

Add a static method initialize() that registers a workmanager periodic task:
  uniqueName: "stira_location_task"
  taskName: "stira_location_check"
  frequency: Duration(minutes: 5)
  constraints: Constraints(networkType: NetworkType.not_required)
  existingWorkPolicy: ExistingWorkPolicy.keep

Add a static method checkLocation() called from the background task. This method:

1. Gets the current device position using Geolocator.getCurrentPosition with
   desiredAccuracy: LocationAccuracy.medium and a timeLimit of 10 seconds.
   If this throws any exception, return silently — location not available.

2. Reads all danger zones and safe zones from stira_prefs. Each zone is a Map
   with fields: "name" (String), "lat" (double), "lng" (double),
   "radius_meters" (double), "is_danger" (bool), "id" (String).

3. For each zone, calculate the distance in meters between the current position
   and the zone's lat/lng using Geolocator.distanceBetween().

4. If distance <= zone radius:
   a. Check last_fired_{zone_id} in stira_prefs. If this timestamp exists and
      is less than 6 hours ago, skip this zone. This prevents spam.
   b. If the zone is_danger is true:
      - Call NotificationService.fireSOSHeartbeat()
      - Wait 2 seconds using Future.delayed(Duration(seconds: 2))
      - Call NotificationService.showNow with:
          id: ID_DANGER_ZONE
          title and body from Messages.dangerZone(zone["name"])
          channelId: "stira_critical"
          payload: "/tools"
      - Increment the zone's trigger_count in stira_prefs by 1.
   c. If the zone is_danger is false (Safe Zone):
      - Call NotificationService.showNow with:
          id: ID_SAFE_ZONE
          title and body from Messages.safeZone(zone["name"])
          channelId: "stira_low"
          payload: "/home"
   d. Write the current timestamp to last_fired_{zone_id} in stira_prefs.

Find main.dart. Add or update the following inside the main() function, before
runApp(). Keep all existing initialization code. Only add what is missing.

Add these calls in this exact order after WidgetsFlutterBinding.ensureInitialized():

  // Must be first — registers the background task handler
  Workmanager().initialize(callbackDispatcher, isInDebugMode: false);

  // Initialize notification channels and local plugin
  await NotificationService.initialize();

Also add this import at the top of main.dart:
  import 'services/stira_intelligence_engine.dart';

call StiraIntelligenceEngine.reactToAction(UserAction.appOpened) so that
returning to the app also cancels stale disengagement notifications.
